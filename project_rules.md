# **Project Rules & Development Standards**

## **1\. Environment & Dependency Management**

* **Tooling:** We use uv as our package manager, but strictly utilizing venv and pip rather than a pyproject.toml or uv directly for project management.  
* **Existing Environment Setup via:** uv venv \--python 3.12 && uv pip install \-r requirements.txt; access via .\venv\Scripts\activate.bat on Windows or source .venv/bin/activate on Unix.

## **2\. Notebook Iteration & Version Control**

* **Jupytext Workflow:** All notebook development must start as .py scripts (using the jupytext light format) to accelerate iteration, linting, and debugging with AI agents.  
* **Syncing:** Scripts are subsequently converted to and tested as .ipynb files. Both .py and .ipynb files may coexist in the repository during active development, but the .py script is the source of truth for code reviews.

## **3\. Architecture & Code Organization**

* **Data Artifacts:** Data must be strictly segregated by type into data/vectors/, data/rasters/, and data/tabular/.  
* **Enumerated Notebooks:** The notebooks/ directory must mirror the data structure (notebooks/vectors/, notebooks/rasters/, notebooks/tabular/). Notebooks should be sequentially numbered (e.g., 01\_fetch\_osm.py) to indicate execution order.  
* **Heavy Machinery Isolation:** Complex data wrangling, custom computer vision extraction logic, and DuckDB spatial indexing must be abstracted into a central utils/ or src/ directory.  
* **Refactoring:** We will periodically refactor bulky notebook code into the utils/ directory to keep notebooks clean and narrative-focused.  
* **End-to-End (E2E) Root Notebook:** The repository root will contain a master E2E notebook. This notebook must be highly narrative-driven, suppressing raw code where possible, to cater to an Urban Planning and policy-oriented audience.

## **4\. Visual Outputs**

* **Map & Figure Generation:** Any critical visual artifacts generated by the notebooks (baseline maps, prediction overlays, demographic charts) must be automatically exported to an outputs/figures/ or outputs/maps/ directory for easy compilation into presentations and slide decks.

## **5\. Data Engineering Preferences**

* **Database Optimization:** Strong preference for local DuckDB database files leveraging the spatial extension. Optimize for vectorized SQL operations and columnar storage over in-memory pandas/geopandas manipulations whenever possible.
